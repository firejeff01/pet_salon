# 事件風暴分析：貳寶寵物美容工作坊定型化契約系統

## Bounded Context 1：飼主管理

### Domain Events（領域事件）
- 🟠 飼主已建立：美容師或客戶填寫飼主基本資料後，系統成功保存飼主資料
- 🟠 飼主已更新：美容師修改飼主資料後，系統成功保存變更
- 🟠 飼主已查詢：美容師依姓名或電話查詢飼主，系統回傳查詢結果
- 🟠 儲值狀態已開啟：美容師為飼主開啟儲值型客戶狀態
- 🟠 儲值狀態已關閉：美容師手動關閉飼主的儲值型客戶狀態
- 🟠 儲值金額已更新：美容師手動修改飼主的儲值總金額

### Commands（命令）
- 🔵 新增飼主：輸入飼主基本資料並保存，由美容師觸發
- 🔵 編輯飼主：修改既有飼主資料並保存，由美容師觸發
- 🔵 查詢飼主：依姓名或電話搜尋飼主，由美容師觸發
- 🔵 開啟儲值狀態：將飼主設為儲值型客戶，由美容師觸發
- 🔵 關閉儲值狀態：將飼主的儲值型客戶狀態關閉，由美容師觸發
- 🔵 更新儲值金額：手動修改飼主儲值總金額，由美容師觸發

### Aggregates（聚合）
- 🟡 飼主（Owner）：管理飼主基本資料、儲值金額、獸醫院指定資訊
  - 接收命令：新增飼主、編輯飼主、查詢飼主、開啟儲值狀態、關閉儲值狀態、更新儲值金額
  - 產生事件：飼主已建立、飼主已更新、飼主已查詢、儲值狀態已開啟、儲值狀態已關閉、儲值金額已更新

### Policies（政策/規則）
- 🔴 飼主姓名、身分證字號、聯絡電話、通訊地址、緊急聯絡人、緊急聯絡人電話、緊急聯絡人與飼主關係為必填
- 🔴 獸醫院資訊未填時，系統自動帶入店家預設（安欣動物醫院、03-3367775、桃園市桃園區中福街60號）
- 🔴 儲值餘額不得小於 0
- 🔴 當儲值金消費至 0 後，美容師需手動關閉該客人的儲值狀態

## Bounded Context 2：寵物管理

### Domain Events（領域事件）
- 🟠 寵物已建立：美容師或客戶填寫寵物資料後，系統成功保存寵物資料
- 🟠 寵物已更新：美容師修改寵物資料後，系統成功保存變更

### Commands（命令）
- 🔵 新增寵物：為飼主新增寵物資料，由美容師觸發
- 🔵 編輯寵物：修改寵物資料並保存，由美容師觸發

### Aggregates（聚合）
- 🟡 寵物（Pet）：管理寵物基本資料、個性、身體狀態檢視、病史，關聯至飼主
  - 接收命令：新增寵物、編輯寵物
  - 產生事件：寵物已建立、寵物已更新

### Policies（政策/規則）
- 🔴 寵物必須關聯到一位飼主
- 🔴 寵物名稱、種類、品種、性別、年齡、是否結育為必填
- 🔴 寵物個性、身體狀態檢視、病史為必填
- 🔴 客戶至少需填寫一隻寵物

## Bounded Context 3：客戶資料填寫

### Domain Events（領域事件）
- 🟠 客戶資料填寫頁已開啟：美容師開啟客戶填寫用畫面
- 🟠 客戶資料已送出：客戶完成飼主與寵物資料填寫並送出

### Commands（命令）
- 🔵 開啟客戶填寫頁：開啟客戶資料填寫畫面，由美容師觸發
- 🔵 送出客戶資料：客戶填寫完成後送出飼主與寵物資料，由客戶觸發

### Aggregates（聚合）
- 🟡 客戶填寫表單（CustomerForm）：管理客戶填寫的飼主與寵物資料
  - 接收命令：開啟客戶填寫頁、送出客戶資料
  - 產生事件：客戶資料填寫頁已開啟、客戶資料已送出

### Policies（政策/規則）
- 🔴 客戶填寫頁不需要登入或帳號
- 🔴 客戶填寫頁不顯示其他客戶資料
- 🔴 客戶填寫頁不提供刪除、瀏覽全部客戶、修改其他預約等店內管理功能
- 🔴 送出前需檢查必填欄位
- 🔴 當 客戶資料已送出 發生時 → 保存飼主資料與寵物資料至 JSON

## Bounded Context 4：預約管理

### Domain Events（領域事件）
- 🟠 預約已建立：美容師成功建立預約
- 🟠 預約已更新：美容師修改預約資料後，系統成功保存變更
- 🟠 預約已取消：美容師將預約標記為取消並填寫取消原因

### Commands（命令）
- 🔵 新增預約：為寵物建立美容預約，由美容師觸發
- 🔵 編輯預約：修改預約日期、時間、狀態與備註，由美容師觸發
- 🔵 取消預約：將預約標記為取消並填寫取消原因，由美容師觸發

### Aggregates（聚合）
- 🟡 預約（Appointment）：管理預約日期、時間、狀態、關聯飼主與寵物
  - 接收命令：新增預約、編輯預約、取消預約
  - 產生事件：預約已建立、預約已更新、預約已取消

### Policies（政策/規則）
- 🔴 預約需包含日期、時間、飼主與寵物
- 🔴 同一飼主的多隻寵物可在同一天建立多筆預約
- 🔴 取消預約時需填寫取消原因
- 🔴 取消後資料不直接刪除
- 🔴 當 預約已建立 發生時 → 行事曆顯示該預約
- 🔴 當 預約已取消 發生時 → 行事曆中已取消預約需有明確標示

## Bounded Context 7：美容紀錄

### Domain Events（領域事件）
- 🟠 美容紀錄已建立：美容師成功保存本次美容服務資料
- 🟠 美容紀錄已更新：美容師修改美容紀錄後成功保存
- 🟠 儲值金額已扣款：儲值型客戶的儲值金額因本次消費而扣除

### Commands（命令）
- 🔵 建立美容紀錄：輸入本次美容服務資料並保存，由美容師觸發
- 🔵 更新美容紀錄：修改美容紀錄並保存，由美容師觸發

### Aggregates（聚合）
- 🟡 美容紀錄（GroomingRecord）：管理單次美容服務的項目、費用、扣款計算
  - 接收命令：建立美容紀錄、更新美容紀錄
  - 產生事件：美容紀錄已建立、美容紀錄已更新、儲值金額已扣款

### Policies（政策/規則）
- 🔴 美容項目為固定選單：洗澡、美容、其他
- 🔴 頁面需自動帶出飼主基本資料與寵物資料（含個性、身體狀態、病史）
- 🔴 若飼主為儲值型客戶且餘額 >= 費用，全額從儲值金扣除
- 🔴 若飼主為儲值型客戶且餘額 < 費用，儲值金全部扣完，差額以現金支付
- 🔴 儲值餘額不得小於 0
- 🔴 當 美容紀錄已建立 發生時 → 資料寫入 grooming-records.json
- 🔴 當 儲值金額已扣款 發生時 → 更新飼主儲值餘額

## Bounded Context 8：契約 PDF 產生

### Domain Events（領域事件）
- 🟠 契約 PDF 已產生：系統成功讀取模板並產生新的契約 PDF
- 🟠 契約版本已建立：同一預約重複產生契約時，建立新版本
- 🟠 契約產生時間已記錄：系統記錄契約產生時間與產生者

### Commands（命令）
- 🔵 產生契約 PDF：依美容紀錄與 PDF 模板重新產生契約 PDF，由美容師觸發
- 🔵 下載契約 PDF：下載或開啟已產生的契約 PDF，由美容師觸發
- 🔵 查看契約版本：查看該筆紀錄的所有契約版本，由美容師觸發

### Aggregates（聚合）
- 🟡 契約（Contract）：管理契約 PDF 的產生、版本與保存
  - 接收命令：產生契約 PDF、下載契約 PDF、查看契約版本
  - 產生事件：契約 PDF 已產生、契約版本已建立、契約產生時間已記錄

### Policies（政策/規則）
- 🔴 PDF 必須以指定契約模板為來源重新產生
- 🔴 原始 PDF 模板不得被修改、覆蓋或移動
- 🔴 第 1 頁與第 4 頁的重複欄位都需套版
- 🔴 第 3 頁簽署日期使用民國年格式
- 🔴 第 4 頁所有欄位都需套版（含個性、身體狀態、病史、獸醫院、儲值金額、結育狀態）
- 🔴 收費金額必須列印在契約 PDF
- 🔴 儲值型客戶帶入扣除本次消費後的剩餘儲值金，非儲值型客戶不帶入
- 🔴 同一預約重複產生契約時產生新版本，不覆蓋原有版本
- 🔴 系統需記錄契約產生時間與產生者
- 🔴 產生契約前需檢查必要資料是否完整
- 🔴 PDF 模板不存在時提示無法產生契約
