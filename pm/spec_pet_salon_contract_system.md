# 貳寶寵物美容工作坊定型化契約系統需求文件

## 1. 文件目的

本文件整理「貳寶寵物美容工作坊定型化契約系統」之產品需求，作為後續系統設計、前後端開發、測試案例撰寫與驗收依據。

系統目標是讓美容師在個人電腦上管理飼主、寵物與預約資料，並於每次美容服務前，依據既有的犬貓美容定型化契約 PDF 模板，自動帶入飼主、寵物與本次美容資訊，重新產生可供客戶簽名的契約 PDF。

## 2. 專案背景

貳寶寵物美容工作坊目前已有一份固定格式的契約文件：

- `C:\WorkSpace\pet_salon\貳寶寵物美容工坊：犬貓美容定型化契約.pdf`

此 PDF 為本系統產生契約時使用的固定模板。系統不需要重新設計契約條款內容，主要工作是每次以此 PDF 模板為基礎，將該次服務需要填寫的資料自動套入模板，再重新生成一份新的契約 PDF，減少人工填寫時間與資料錯誤。

經初步檢視，該 PDF 共 4 頁，未含可直接填寫的 AcroForm 表單欄位。因此系統產生契約時，需求上應視為「讀取原始 PDF 模板、在輸出副本上套入文字與勾選標記、重新產生新的 PDF 檔案」，而不是填入 PDF 表單欄位，也不是直接修改原始模板檔案。

PDF 模板共 4 頁，第 1-3 頁為定型化契約條款，第 4 頁為「寵物美容契約書」表格。第 1 頁與第 4 頁均包含需要填寫的資料欄位，兩頁都需要套版。

## 3. 系統目標

1. 建立可在個人電腦使用的寵物美容契約產生系統。
2. 支援飼主基本資料新增、查詢、編輯與保存。
3. 支援寵物資料新增、查詢、編輯與保存，並關聯到飼主。
4. 支援行事曆檢視預約資訊（月、週、日檢視）。
5. 點擊行事曆上的預約資訊後，可開啟該日期的預約列表。
6. 從當日預約列表選擇單一寵物後，可輸入本次美容服務相關資料。
7. 按下「產生契約」後，自動產出本次服務的契約 PDF。
8. 契約 PDF 必須自動帶入飼主資料、寵物資料與本次美容服務資料，涵蓋第 1 頁與第 4 頁所有需填寫欄位。
9. 契約 PDF 需支援飼主在電腦上電子簽名。
10. 系統資料以 JSON 檔案保存，不使用資料庫。
11. 支援飼主儲值金額管理。
12. 支援資料匯出與一鍵備份。
13. 系統需封裝為可雙擊啟動的桌面應用程式，同時保留 npm scripts 作為開發入口。

## 4. 使用者與角色

本系統不設計帳號、密碼、登入流程或角色權限控管。美容師與客戶皆可在已開啟的本機系統畫面中直接操作指定功能。

### 4.1 美容師 / 店家操作人員

主要系統使用者，負責：

- 建立與維護飼主資料。
- 建立與維護寵物資料。
- 管理飼主儲值金額。
- 開啟客戶資料填寫畫面，讓客戶直接填寫飼主與寵物資料。
- 建立與查看預約。
- 輸入每次美容服務資訊。
- 產生契約 PDF。
- 將 PDF 提供給飼主在電腦上簽名。

### 4.2 飼主 / 客戶

飼主不需要登入系統。飼主可由美容師開啟資料填寫畫面後，直接在系統上填寫或確認資料。飼主會：

- 提供個人基本資料。
- 提供寵物資料。
- 直接在系統畫面填寫或確認飼主與寵物資料。
- 確認美容服務內容。
- 在電腦上簽署系統產生的契約 PDF。

## 6. 技術需求

### 6.0 系統架構

- 系統採前後端分離架構。
- 前端使用 Vue.js，負責畫面呈現、表單互動、前端欄位檢查與呼叫 API。
- 前端不得直接讀寫 JSON 檔案、不得直接操作 PDF 模板、不得直接產生契約 PDF。
- 後端使用 Node.js 建立 API Server，負責商業邏輯、資料驗證、JSON 檔案讀寫與 PDF 產生。
- 前端所有資料查詢、資料保存、預約操作、美容紀錄操作與契約產生，都必須透過後端 API 完成。
- 前後端需可分別啟動、分別測試。
- 開發階段前後端可分別啟動，例如 Vue 開發伺服器與 Node.js API Server 使用不同 localhost port。
- 正式本機使用時，前後端可部署在同一個站台，由 Node.js API Server 提供 Vue build 後的靜態檔案。
- 正式本機站台中，後端 API 統一使用 `/api` 路徑，其他非 `/api` 路徑由 Vue 前端路由處理。
- Swagger / OpenAPI 文件站台可在同一個站台提供，例如 `/api-docs`。
- 同站台部署不代表前後端混合，前端仍不得直接讀寫 JSON 檔案或產生 PDF，所有資料操作仍需透過 `/api`。

### 6.1 前端

- 使用 Vue.js。
- 提供桌面瀏覽器操作介面。
- 不需要登入頁、帳號管理頁或權限設定頁。
- 前端純粹呼叫後端 API，不直接存取本機資料檔案。
- 前端需集中管理 API 呼叫邏輯，例如建立 API client 或 service module。
- 前端需處理 API 回傳的成功、失敗、驗證錯誤與 PDF 下載結果。
- 前端需提供電子簽名元件，供飼主在電腦上簽名。
- 主要畫面包含：
  - 首頁 / 今日預約總覽。
  - 飼主管理（含儲值金額管理）。
  - 寵物管理。
  - 客戶資料填寫頁。
  - 行事曆（月、週、日檢視）。
  - 當日預約列表。
  - 美容服務資料輸入頁。
  - 電子簽名頁。
  - 契約 PDF 產生結果頁或下載提示。
  - 資料備份頁。

### 6.2 後端

- 使用 Node.js。
- 建立三層式架構的 REST API Server，提供 API 給前端呼叫。
- API Server 需提供 Swagger / OpenAPI 文件站台，供開發者查看與測試 API。
- 負責 JSON 檔案讀寫。
- 負責讀取 PDF 模板、套入資料並輸出新的 PDF 檔案。
- 負責輸出檔案命名與保存。
- 負責資料匯出與備份功能。
- 後端程式需可在 Windows 與 macOS 本機環境執行。
- 檔案路徑處理需使用 Node.js `path` 相關 API，不得將 Windows 路徑分隔符 `\` 或 macOS / Unix 路徑分隔符 `/` 寫死在程式邏輯中。

### 6.3 後端三層式架構

後端需採三層式架構，至少包含以下分層：

#### 6.3.1 Controller / Route Layer

- 負責定義 API 路由。
- 負責接收 HTTP request。
- 負責基本 request 格式檢查。
- 負責呼叫 Service Layer。
- 負責回傳 HTTP response。
- 不應直接讀寫 JSON 檔案。
- 不應直接產生 PDF。

#### 6.3.2 Service / Business Logic Layer

- 負責主要商業邏輯。
- 負責欄位驗證與流程規則。
- 負責處理飼主、寵物、預約、美容紀錄與契約產生的 use case。
- 負責儲值金額計算與驗證邏輯。
- 負責協調 Repository Layer 與 PDF 產生模組。
- 不應直接處理 HTTP request / response 物件。

#### 6.3.3 Repository / Data Access Layer

- 負責 JSON 檔案讀寫。
- 負責資料查詢、建立、更新與關聯資料存取。
- 負責處理 JSON 檔案不存在、格式錯誤、寫入失敗等資料層問題。
- 不應包含畫面流程或 HTTP response 邏輯。

#### 6.3.4 PDF Generator Module

- 可作為 Service Layer 呼叫的獨立模組。
- 負責讀取 PDF 模板。
- 負責將資料套入輸出 PDF（第 1 頁與第 4 頁所有需填寫欄位）。
- 負責輸出新的 PDF 檔案。
- 負責將電子簽名圖片嵌入 PDF 指定位置。
- 不應直接處理前端 request。

### 6.4 Swagger / OpenAPI 文件站台

- 後端 API Server 需提供 Swagger / OpenAPI 文件站台。
- Swagger 站台需可在本機瀏覽器開啟。
- 建議 Swagger UI 路徑為 `/api-docs`。
- 建議 OpenAPI JSON 路徑為 `/api-docs.json`。
- Swagger 文件需涵蓋所有公開 API。
- Swagger 文件需包含：
  - API 路徑。
  - HTTP method。
  - request body schema。
  - query string / path parameter。
  - response schema。
  - 常見錯誤 response。
- Swagger UI 需可直接測試 API。
- 因系統不需要登入，Swagger 測試 API 時不需設定登入 token。
- Swagger 文件需隨 API 變更同步更新。
- Swagger 站台僅供本機開發、測試與交付驗收使用，不應開放到公開網路。

### 6.5 資料儲存

- 不使用資料庫。
- 使用 JSON 檔案保存系統資料。
- JSON 檔案預設放在桌面的專屬資料夾，例如：`桌面/貳寶寵物美容工坊資料/`。
- 使用者可在系統設定中自行變更資料目錄路徑。
- 預設資料目錄結構：
  - `data/owners.json`
  - `data/pets.json`
  - `data/appointments.json`
  - `data/grooming-records.json`
  - `data/settings.json`
- PDF 產出檔案可保存於：
  - `generated-contracts/`

### 6.6 PDF 模板與產生

- 系統需將使用者提供的既有 PDF 作為契約模板。
- 原始 PDF 模板檔案需視為唯讀來源，不得在產生契約時直接修改或覆蓋。
- 每次產生契約時，系統需讀取 PDF 模板，將飼主、寵物與本次美容資料套入，並重新生成一份新的 PDF 檔案。
- 因模板未含可填寫欄位，系統可使用 PDF 疊字與勾選標記方式，將資料放到輸出 PDF 的指定位置。
- 第 1 頁與第 4 頁均包含需要填寫的欄位，兩頁都需要套版。
- 第 1 頁與第 4 頁有重複欄位（飼主姓名、電話、寵物名稱等），兩頁都要套版。
- 第 3 頁簽署日期需系統自動帶入，使用民國年格式（例如：中華民國 115 年 4 月 16 日）。
- 產出 PDF 不得破壞原契約條款內容。
- 產出 PDF 應可下載、列印，並可供飼主簽名。
- 同一筆預約重複產生契約時，系統產生新版本檔案，不覆蓋原有版本。
- PDF 模板路徑需可設定，不能只支援固定 Windows 路徑 `C:\WorkSpace\pet_salon\...`。
- 系統需支援在 Windows 與 macOS 上開啟或下載產生後的 PDF。

### 6.7 電子簽名

- 前端需提供簽名板元件，讓飼主在電腦上手寫簽名。
- 簽名圖片需傳送至後端，由 PDF Generator 嵌入契約 PDF 的甲方簽章位置。
- 第 3 頁甲方簽章與第 4 頁飼主簽名欄位均需嵌入簽名圖片。
- 本系統不要求電子簽章法規等級的簽名驗證。

### 6.8 跨平台本機運行

系統預期在店家個人電腦本機執行，且可能部署於 Windows 或 macOS，因此需符合以下要求：

- 前端與後端啟動方式需支援 Windows PowerShell / Command Prompt 與 macOS Terminal。
- 專案安裝與啟動指令應盡量使用跨平台 npm scripts，例如 `npm install`、`npm run dev`、`npm run start`。
- 不得依賴只存在於單一作業系統的 shell 指令作為核心流程。
- 本機服務預設應綁定 `localhost`，避免未授權的區域網路存取。
- 若需開啟檔案總管或 PDF 檢視器，需依作業系統分別處理 Windows `start`、macOS `open` 等差異，或改以瀏覽器下載方式降低平台差異。
- 所有可設定路徑應支援相對路徑與絕對路徑。
- 應避免使用檔名不合法字元，尤其是 Windows 不允許的 `\ / : * ? " < > |`。
- 產生的 PDF 檔名需使用跨平台安全格式，例如 `YYYYMMDD_HHmm_寵物名稱_飼主姓名_契約.pdf`，並處理特殊字元替換。
- 同一筆預約重複產生契約時，檔名需加上版本編號，例如 `YYYYMMDD_HHmm_寵物名稱_飼主姓名_契約_v2.pdf`。
- 中文檔名與中文字型需在 Windows 與 macOS 上測試。
- PDF 模板套版使用的中文字型需隨專案提供或明確設定，避免不同作業系統缺字、字型寬度不同或 PDF 顯示異常。

### 6.9 桌面應用程式封裝

- 系統需封裝為可雙擊啟動的桌面應用程式，Windows 與 macOS 均需提供。
- 封裝後使用者不需要手動安裝 Node.js 或執行 npm 指令。
- 封裝方式可使用 Electron 或其他適合的桌面應用程式框架。
- 啟動後自動開啟瀏覽器或內建視窗載入系統介面。
- 封裝不得成為唯一啟動方式，仍需保留 npm scripts 作為開發入口。

### 6.10 測試需求

前端與後端需分別撰寫單元測試，並能分別執行。

#### 6.10.1 前端單元測試

- 前端需針對主要 Vue 元件、表單驗證、API client / service module 撰寫單元測試。
- 前端測試需 mock 後端 API，不應依賴實際後端服務啟動。
- 前端測試至少需涵蓋：
  - 飼主資料表單必填檢查。
  - 寵物資料表單必填檢查。
  - 客戶資料填寫頁送出流程。
  - 行事曆預約資料顯示邏輯（月、週、日檢視）。
  - 當日預約列表排序與顯示邏輯。
  - 美容服務資料送出與產生契約按鈕狀態。
  - 電子簽名元件基本功能。
  - 儲值金額顯示與扣款計算。
  - API 成功、失敗與驗證錯誤回應的畫面處理。

#### 6.10.2 後端單元測試

- 後端需針對 Controller、Service、Repository 與 PDF Generator 分別撰寫測試。
- Controller 測試需驗證 API request / response、HTTP status code 與錯誤格式。
- Service 測試需驗證商業邏輯、必填欄位、關聯資料、儲值金額計算與契約產生流程。
- Repository 測試需驗證 JSON 檔案讀寫、資料查詢、資料更新、檔案不存在與 JSON 格式錯誤處理。
- PDF Generator 測試需驗證會讀取模板、輸出新 PDF、且不修改原始 PDF 模板。
- 後端單元測試不應依賴正式資料檔，需使用測試用暫存資料目錄或 mock repository。

#### 6.10.3 測試執行

- 前端需提供單元測試指令，例如 `npm run test:unit`。
- 後端需提供單元測試指令，例如 `npm run test:unit`。
- 若前後端位於同一 repository，需提供可分別執行前端與後端測試的指令。
- 測試需能在 Windows 與 macOS 執行。
- 測試資料不得覆寫正式 JSON 資料或正式產生的契約 PDF。

- 儲值餘額不得小於 0。
- 當儲值金消費至 0 後，美容師需手動關閉該客人的儲值狀態。
- 非儲值型客戶，契約上儲值金額欄位不帶入或顯示為空。

獸醫院資訊規則：

- 飼主可填寫自己指定的獸醫院名稱、電話、地址。
- 若飼主未填，系統自動帶入店家預設的獸醫院資訊（安欣動物醫院、03-3367775、桃園市桃園區中福街 60 號）。

## 8. 功能需求

### 8.1 飼主管理

#### 8.1.1 新增飼主

美容師可以建立新飼主資料。美容師也可以開啟客戶填寫畫面，讓飼主直接在系統上填寫自己的基本資料。

驗收條件：

- 輸入必要欄位後可成功保存。
- 系統需自動產生飼主 ID。
- 保存後資料需寫入 JSON 檔案。
- 保存後可於飼主列表查詢到該筆資料。
- 客戶填寫資料不需要帳號、密碼或登入流程。
- 客戶送出資料後，系統需將資料保存為飼主資料，或進入美容師確認後保存的流程。

#### 8.1.2 編輯飼主

美容師可以修改既有飼主資料。已產生契約後仍可修改飼主資料，並允許重新產生契約。

驗收條件：

- 可從飼主列表或搜尋結果開啟編輯。
- 修改後可保存。
- 保存後 JSON 檔案內容同步更新。
- 與該飼主關聯的寵物與預約仍維持關聯。

#### 8.1.3 查詢飼主

美容師可以依姓名或電話查詢飼主。

驗收條件：

- 輸入姓名關鍵字可查詢。
- 輸入電話關鍵字可查詢。
- 查詢結果需顯示飼主姓名、電話及寵物摘要。

#### 8.1.4 客戶填寫飼主資料

美容師可以從系統開啟一個客戶填寫用畫面，交由飼主直接輸入基本資料。此畫面不需要登入，也不需要客戶建立帳號。

驗收條件：

- 美容師可從系統開啟客戶填寫畫面。
- 客戶可直接輸入姓名、身分證字號、聯絡電話、通訊地址、緊急聯絡人、緊急聯絡人電話、緊急聯絡人與飼主關係與指定動物醫療機構（含電話、地址）。
- 客戶送出前，系統需檢查必要欄位。
- 客戶送出後，資料需保存至 JSON，或進入待美容師確認狀態。
- 送出完成後，畫面不應顯示其他客戶資料。
- 客戶填寫畫面需避免提供刪除、瀏覽全部客戶、修改其他預約等店內管理功能。

#### 8.1.5 儲值金額管理

美容師可以管理飼主的儲值金額。

驗收條件：

- 美容師可在飼主資料中開啟或關閉儲值型客戶狀態。
- 美容師可在收到飼主儲值金額後，手動修改儲值總金額。
- 儲值餘額不得小於 0。
- 儲值金額變更需記錄於飼主資料中。

### 8.2 寵物管理

#### 8.2.1 新增寵物

美容師可以為既有飼主新增寵物資料。美容師也可以讓飼主在客戶填寫畫面直接輸入寵物資料。

驗收條件：

- 寵物必須關聯到一位飼主。
- 寵物名稱、種類、品種、性別、年齡、是否結育需可保存。
- 保存後可於飼主詳細頁或寵物列表看到該寵物。
- 客戶填寫寵物資料不需要登入。
- 客戶送出的寵物資料需關聯到該次填寫的飼主資料。

#### 8.2.2 客戶填寫寵物資料

美容師可以在客戶填寫流程中讓飼主輸入一隻或多隻寵物資料。

驗收條件：

- 客戶可輸入寵物名稱、種類、品種、性別、年齡、是否結育、晶片號碼、未登記身分識別方式、寵物個性（多選勾選）、寵物身體狀態檢視、寵物病史（多選勾選）。
- 客戶至少需填寫一隻寵物。
- 客戶送出後，寵物資料需與飼主資料建立關聯。
- 送出完成後，美容師可在飼主詳細頁看到客戶填寫的寵物資料。

#### 8.2.3 編輯寵物

美容師可以修改寵物基本資料。已產生契約後仍可修改寵物資料，並允許重新產生契約。

驗收條件：

- 可修改品種、性別、年齡、是否結育、晶片號碼、寵物個性、身體狀態、病史等欄位。
- 修改後資料需保存於 JSON。
- 既有預約與美容紀錄仍需維持關聯。

### 8.3 預約管理

#### 8.3.1 新增預約

美容師可以為寵物建立美容預約。

驗收條件：

- 預約需包含日期、時間、飼主與寵物。
- 建立後需顯示於行事曆。
- 同一寵物可有多筆不同日期的預約。
- 同一飼主的多隻寵物可在同一天建立多筆預約。

#### 8.3.2 編輯預約

美容師可以調整預約日期、時間、狀態與備註。已產生契約後仍可修改預約資料，並允許重新產生契約。

驗收條件：

- 修改後行事曆需反映最新資料。

#### 8.3.3 取消預約

美容師可以將預約標記為取消，並需填寫取消原因。

驗收條件：

- 取消後資料不應直接刪除。
- 取消時需填寫取消原因。
- 行事曆中已取消預約需有明確標示，可選擇顯示或隱藏。

### 8.4 行事曆檢視

系統需提供行事曆功能，用於顯示預約資訊。

驗收條件：

- 支援月檢視、週檢視與日檢視三種模式。
- 可切換不同檢視模式。
- 每個日期需顯示該日預約摘要，例如預約數或寵物名稱。
- 點擊日期或日期上的預約資訊後，需開啟該日預約列表。
- 當日預約列表需依時間排序。

### 8.5 當日預約列表

當美容師點擊行事曆中的日期或預約資訊後，系統需顯示該日所有預約。

列表建議顯示：

- 預約時間。
- 寵物名稱。
- 飼主姓名。
- 聯絡電話。
- 預約狀態。
- 是否已產生契約。

驗收條件：

- 點擊單一預約或寵物可進入本次美容服務資料輸入頁。
- 已取消預約需有明確狀態標示，並顯示取消原因。
- 已產生契約的預約需能查看或下載既有 PDF。

### 8.6 本次美容服務資料輸入

美容師從當日預約列表選擇單一寵物後，可輸入本次美容資訊。

驗收條件：

- 頁面需自動帶出飼主基本資料。
- 頁面需自動帶出寵物基本資料（含個性、身體狀態、病史）。
- 美容師可選擇本次美容項目（固定選單：洗澡、美容、其他）並輸入各項費用。
- 美容師可確認或更新寵物身體狀態檢視、病史與個性。
- 若飼主為儲值型客戶，自動計算儲值金額扣款與現金支付金額。
- 按下保存後，資料需寫入 `grooming-records.json`。
- 按下產生契約前，系統需檢查必要資料是否完整。

## 9. 建議 API 需求

實際路由可於技術設計階段調整，但需涵蓋下列能力。所有 API 需由後端三層式架構提供，前端只能透過 API 操作資料，不得直接讀寫 JSON 檔案或產生 PDF。

API 設計原則：

- API 回應格式需一致。
- API 錯誤格式需一致。
- 必填欄位缺漏需回傳明確錯誤訊息。
- 查無資料需回傳明確狀態。
- Controller 只負責 HTTP 介面，不直接處理 JSON 檔案或 PDF 產生。
- Service 負責商業流程。
- Repository 負責資料讀寫。
- API 需產出 OpenAPI 規格並提供 Swagger UI 測試頁。
- Swagger UI 需可測試新增、查詢、更新、產生契約等 API。

### 9.1 飼主 API

- `GET /api/owners`：取得飼主列表。
- `GET /api/owners/:ownerId`：取得單一飼主。
- `POST /api/owners`：新增飼主。
- `PUT /api/owners/:ownerId`：更新飼主（含儲值金額）。

### 9.2 寵物 API

- `GET /api/pets`：取得寵物列表。
- `GET /api/pets/:petId`：取得單一寵物。
- `GET /api/owners/:ownerId/pets`：取得飼主的寵物。
- `POST /api/pets`：新增寵物。
- `PUT /api/pets/:petId`：更新寵物。

### 9.3 預約 API

- `GET /api/appointments`：取得預約列表，可依日期區間查詢。
- `GET /api/appointments/date/:date`：取得指定日期預約。
- `POST /api/appointments`：新增預約。
- `PUT /api/appointments/:appointmentId`：更新預約（含取消與取消原因）。

### 9.4 美容紀錄 API

- `GET /api/grooming-records/:appointmentId`：取得預約對應美容紀錄。
- `POST /api/grooming-records`：建立美容紀錄。
- `PUT /api/grooming-records/:recordId`：更新美容紀錄。

### 9.5 契約 API

- `POST /api/contracts/generate`：依美容紀錄與指定 PDF 模板重新產生契約 PDF（產生新版本）。
- `GET /api/contracts/:recordId`：下載或開啟已產生契約 PDF。
- `GET /api/contracts/:recordId/versions`：取得該筆紀錄的所有契約版本。
- `POST /api/contracts/:recordId/sign`：上傳電子簽名並嵌入 PDF。

### 9.6 備份 API

- `POST /api/backup`：執行一鍵備份。
- `GET /api/backup/list`：取得備份檔案列表。

## 10. JSON 儲存需求

### 10.1 基本原則

- 每一種主要資料使用獨立 JSON 檔案。
- 寫入 JSON 時需避免資料毀損。
- 建議寫入前先備份原檔，或採用暫存檔寫入完成後再替換正式檔。
- 所有資料需包含穩定 ID，不應以陣列位置作為關聯依據。
- 時間格式建議使用 ISO 8601 字串。
- JSON 檔案需使用 UTF-8 編碼，以確保 Windows 與 macOS 都能正確保存中文資料。
- JSON 讀寫需使用跨平台路徑處理，不得依賴固定磁碟代號或固定使用者目錄。

## 12. 非功能需求

### 12.1 使用環境

- 系統主要在個人電腦上使用。
- 預期由店家內部人員操作。
- 前後端可在同一台電腦執行。
- 系統需支援 Windows 與 macOS 本機環境，兩者均為正式交付環境。
- 系統不可假設固定磁碟代號，例如 `C:\`。
- 系統不可假設固定專案根目錄，例如 `C:\WorkSpace\pet_salon`。
- README 或交付文件需分別說明 Windows 與 macOS 的安裝、啟動、資料備份與 PDF 模板設定方式。

### 12.2 部署與啟動

- 系統需封裝為可雙擊啟動的桌面應用程式，Windows 與 macOS 均需提供。
- 封裝後使用者不需要手動安裝 Node.js 或執行 npm 指令。
- 最低要求需能透過 Node.js 與 npm 在 Windows / macOS 啟動（開發入口）。
- 開發階段可分別啟動前端與後端，例如：
  - 前端 Vue dev server：`http://localhost:5173`
  - 後端 Node.js API Server：`http://localhost:3000`
- 正式本機使用時，建議由 Node.js API Server 同時提供前端靜態檔案與後端 API，讓使用者只需要開啟單一站台，例如 `http://localhost:3000`。
- 正式本機站台路由規則：
  - `/api/*`：由後端 API Server 處理。
  - `/api-docs`：顯示 Swagger UI。
  - `/api-docs.json`：提供 OpenAPI JSON。
  - 其他路徑：回傳 Vue build 後的前端頁面，交由 Vue Router 處理。
- 前端在正式本機部署時，應以相對路徑呼叫 API，例如呼叫 `/api/owners`，避免寫死不同 port 或不同主機名稱。
- 封裝不得成為唯一啟動方式，仍需保留 npm scripts 作為跨平台標準開發入口。

### 12.3 效能

- 一般查詢與保存操作應於 1 秒內完成。
- PDF 產生應於合理時間內完成，建議 5 秒內。

### 12.4 可維護性

- PDF 欄位座標應集中管理，避免散落在程式碼各處。
- 店家固定資料，如商號名稱、電話、代表人、地址，應可集中設定。
- 店家預設獸醫院資訊（安欣動物醫院、03-3367775、桃園市桃園區中福街 60 號）應可集中設定。
- JSON 檔案格式應清楚，方便必要時人工檢查。
- PDF 模板路徑、資料目錄、輸出目錄、中文字型路徑應集中於設定檔管理。
- 設定檔應以相對路徑為優先，並允許使用者改成絕對路徑。

### 12.5 資料安全

- 因系統保存飼主個資，資料目錄不應放在公開網路資料夾。
- 系統不設計帳號、密碼或登入流程，因此本機服務預設應限制在 `localhost` 使用。
- 若需讓客戶填寫資料，應由美容師在現場開啟客戶資料填寫頁，不應開放成公開網路表單。
- 客戶資料填寫頁需限制可操作範圍，避免客戶看到或修改其他客戶資料。
- 系統需提供一鍵備份功能，Windows 與 macOS 各自提供。
- 備份檔案需包含時間戳記，避免覆蓋舊備份。
- Windows 與 macOS 的資料備份說明需明確標示實際資料夾位置。

### 12.6 錯誤處理

系統需處理下列情境：

- JSON 檔案不存在時，自動建立空資料檔。
- JSON 檔案格式錯誤時，提示使用者並避免覆寫原檔。
- PDF 模板不存在時，提示無法產生契約。
- PDF 產生失敗時，保留美容紀錄並顯示錯誤訊息。
- 必填欄位未填時，不允許產生契約。
- 資料目錄或輸出目錄沒有寫入權限時，需提示使用者調整目錄或權限。
- PDF 模板路徑在不同作業系統無效時，需提示使用者重新設定模板位置。
- 儲值餘額不足時，需正確計算儲值金扣款與現金支付金額，儲值餘額不得小於 0。

## 14. 驗收標準

### 14.1 飼主與寵物資料

- 可新增一位飼主（含身分證字號、通訊地址、緊急聯絡人、緊急聯絡人與飼主關係）。
- 可為該飼主新增至少一隻寵物（含品種、性別、年齡、是否結育、個性、身體狀態、病史）。
- 可編輯飼主與寵物資料。
- 已產生契約後仍可修改資料並重新產生契約。
- 關聯資料不因編輯而遺失。
- 可由美容師開啟客戶資料填寫頁，讓飼主直接填寫飼主與寵物資料。
- 客戶填寫資料全程不需要帳號、密碼或登入。
- 客戶送出資料後，美容師可在系統中看到該筆飼主與寵物資料。

### 14.2 儲值金額

- 可為飼主開啟儲值型客戶狀態。
- 美容師可手動修改儲值金額。
- 產生契約時正確計算儲值金額扣款。
- 儲值餘額不足時，差額以現金支付。
- 儲值餘額不得小於 0。
- 非儲值型客戶，契約上儲值金額欄位不帶入。

### 14.3 預約與行事曆

- 可建立一筆預約。
- 同一飼主的多隻寵物可在同日建立多筆預約。
- 預約會出現在正確日期的行事曆中。
- 行事曆支援月、週、日三種檢視模式。
- 點擊該日期可看到當日預約列表。
- 當日列表可點擊該寵物進入美容服務資料頁。
- 取消預約時需填寫取消原因。

### 14.4 美容資料與契約產生

- 可輸入本次美容資料（固定選單：洗澡、美容、其他，各項費用）。
- 可保存美容資料。
- 可按下產生契約。
- 系統會產出一份 PDF。
- PDF 第 1 頁與第 4 頁均有帶入該飼主、該寵物與本次美容資料。
- 第 4 頁寵物個性、身體狀態檢視、寵物病史均正確勾選。
- 收費金額有列印在契約 PDF。
- 儲值金額正確帶入（儲值型客戶）。
- PDF 使用指定模板重新產生，且原始模板不被修改。
- 同一預約重複產生契約時，產生新版本不覆蓋原有版本。
- 系統記錄契約產生時間與產生者。
- PDF 可開啟與列印。

### 14.5 電子簽名

- 飼主可在電腦上手寫簽名。
- 簽名可清除重簽。
- 簽名圖片正確嵌入 PDF 第 3 頁甲方簽章與第 4 頁飼主簽名位置。
- 第 3 頁簽署日期正確帶入民國年格式。

### 14.6 JSON 儲存

- 新增、編輯資料後，JSON 檔案內容會更新。
- 關閉系統後重新啟動，既有資料仍可讀取。
- 系統不依賴資料庫服務。
- 資料目錄預設在桌面專屬資料夾，可由使用者自行變更。

### 14.7 資料備份

- 一鍵備份功能可正常執行。
- 備份檔案包含所有 JSON 資料與已產生的契約 PDF。
- 備份檔案包含時間戳記。
- Windows 與 macOS 均可執行備份。

### 14.8 架構與測試

- 前端不得直接讀寫 JSON 檔案。
- 前端不得直接操作 PDF 模板或產生 PDF。
- 前端需透過後端 API 完成資料查詢、資料保存與契約產生。
- 開發階段前端與後端可分別啟動。
- 正式本機使用時，前端與後端可在同一個站台提供服務。
- 正式本機站台需能透過單一網址開啟前端頁面，例如 `http://localhost:3000`。
- 正式本機站台的 API 需統一位於 `/api` 路徑底下。
- 正式本機站台需可開啟 Swagger UI，例如 `http://localhost:3000/api-docs`。
- Swagger UI 需能直接測試 API。
- OpenAPI JSON 需可透過指定路徑取得，例如 `http://localhost:3000/api-docs.json`。
- 後端需符合 Controller、Service、Repository 三層式架構。
- 後端 PDF 產生需由獨立 PDF Generator 模組或等效模組負責。
- 前端單元測試可獨立執行。
- 後端單元測試可獨立執行。
- 前後端測試不得覆寫正式 JSON 資料與正式契約 PDF。
- Windows 與 macOS 均可執行前端與後端測試。

### 14.9 桌面應用程式

- 系統可封裝為可雙擊啟動的桌面應用程式。
- Windows 與 macOS 均提供封裝版本。
- 使用者不需要手動安裝 Node.js 或執行 npm 指令即可使用。

## 17. 成功情境範例

1. 美容師新增飼主「王小明」，填入姓名、身分證字號、電話、地址、緊急聯絡人（含與飼主關係）。
2. 美容師替王小明新增寵物「Lucky」，填入品種、性別、年齡、是否結育、個性、身體狀態與病史。
3. 美容師建立 Lucky 在 2026-04-20 14:00 的美容預約。
4. 美容師在行事曆上（月/週/日檢視）看到 2026-04-20 有預約。
5. 美容師點擊該日預約資訊，開啟當日預約列表。
6. 美容師點擊 Lucky 的預約。
7. 系統開啟本次美容資料頁，並自動帶出王小明與 Lucky 的資料（含個性、身體狀態、病史）。
8. 美容師選擇美容項目（洗澡）、輸入費用，系統自動計算儲值金額扣款。
9. 美容師按下「產生契約」。
10. 系統讀取指定 PDF 模板，套入資料後重新產生一份契約 PDF（第 1 頁與第 4 頁均帶入資料）。
11. 系統記錄契約產生時間與產生者。
12. 美容師開啟電子簽名畫面，王小明在電腦上簽名。
13. 系統將簽名嵌入 PDF 第 3 頁甲方簽章與第 4 頁飼主簽名位置。
14. 美容師開啟 PDF，確認資料已帶入（含簽名、民國年日期），列印存檔。
